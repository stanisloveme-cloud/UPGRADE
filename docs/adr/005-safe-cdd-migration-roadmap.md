# ADR 005: Стратегия безопасной миграции на CDD (Contract-Driven Development)

## Статус
Утверждено (В процессе реализации)

## Контекст (Context)
Внедрение строгих контрактов API (CDD) и автотестирования (Playwright API + UI) несет риск поломки текущего функционального кода и блокировки работы продакшен-ветки. Требуется поэтапный план перевода "недодокументированного" хардкода на автосгенерированные хуки.

## Решение (Decision)
Принята стратегия **Strangler Fig (Паттерн Душителя)** для постепенной миграции бизнес-логики на `Dev` стенде без отказов:

1. **Этап 0 (Безрисковый):** Реструктуризация документации, изоляция ИИ-промптов в `.agent/prompts/`, создание ADR. (Выполняется прямо сейчас).
2. **Этап 1 (Подготовка генерации):** Внедрение OpenAPI/Swagger конфигурации на стороне NestJS.
3. **Этап 2 (Параллельный CDD):** Внедрение кодогенератора **Orval** на фронтенде (`src/api-generated/`). Старые вызовы `axios/fetch` **не удаляются**.
4. **Этап 3 (Покомпонентная миграция):** Постепенный перевод модулей (напр., "Brands") на автогенерируемые хуки. Покомпонентно, в изолированных ветках `feature/cdd-module-name`.
5. **Этап 4 (Мягкие Автотесты):** Написание API-ориентированных тестов Playwright параллельно с существующими UI-тестами.
6. **Этап 5 (Отсечение и Strict Mode):** Когда 100% приложения переведено, включается "Жесткий режим" CI/CD в ветке `dev`: билд падает при любом рассинхроне TS-типов и JSON Schema.

## Последствия (Consequences)
**Позитивные:** Исключается риск простоя `Dev` стенда. Текущая продуктовая разработка не блокируется. 
**Ограничения:** В течение 2-3 недель в проекте будут существовать два подхода работы с сетью одновременно (Legacy Axios и Generated Hooks).
