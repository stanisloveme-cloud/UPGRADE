# ADR 002: Ограничения NestJS Validation Pipe

## Статус
Утверждено (Реализовано)

## Контекст (Context)
При разработке UI на базе React + Ant Design, таблицы и формы (`ProTable`, `ProFormList`) автоматически встраивают в payload скрытые системные поля (например, `_action`, индексы, или служебное поле `id` для анти-дубликатора). 
Изначально глобальный `ValidationPipe` в NestJS на бэкенде был настроен с флагом `forbidNonWhitelisted: true`. Это приводило к тому, что бэкенд возвращал жесткую ошибку `400 Bad Request` при получении этих скрытых UI-свойств, в результате чего модалки падали или зависали.

## Решение (Decision)
Отказаться от `forbidNonWhitelisted: true`.
Использовать только `whitelist: true` в глобальной конфигурации `ValidationPipe` NestJS.

## Последствия (Consequences)
**Позитивные:** Бэкенд теперь безопасно и бесшумно обрезает (strip) неизвестные поля, отправляемые Ant Design, принимая только те, что жестко описаны в DTO. Взаимодействие фронта и бэка стабилизировалось.
**Отрицательные:** Нельзя строго наказывать фронтенд за отправку лишнего "мусорного" JSON'а на уровне валидатора (запрос все равно пройдет, мусор просто отсечется). Это возлагает бóльшую ответственность на Contract-Driven Development (CDD). Предписание задокументировано в `.antigravityrules`.
