# PRD-11: Библиотека Брендов, Интеграция с Tilda и Улучшения UX

## 1. Обзор
Это обновление представляет глобальную библиотеку "Бренды" (ранее "Проверка брендов"), улучшая управление спонсорами и партнерами на разных мероприятиях. В документе также описаны необходимые API и флаги данных для интеграции расписания, спикеров и спонсоров из CRM на внешний сайт Tilda. Наконец, включены улучшения UX, такие как автоматические черно-белые фильтры для фотографий спикеров и пиксель-пёрфект (pixel-perfect) верстка "Памятки спикера".

## 2. Требования

### 2.1 Библиотека Брендов (Глобальные Спонсоры)
- **Переименование раздела**: "Проверка брендов" -> "Бренды".
- **Глобальное хранение**: Бренды хранятся глобально, независимо от конкретного мероприятия, сохраняя исторические данные об их участии.
- **Обновления карточки**:
  - **Добавить**: "Сегменты рынка" - множественный выбор из предопределенного рубрикатора.
  - **Добавить**: "Выгружать на сайт" - чекбокс (boolean) для управления публичной видимостью.
  - **Изменить**: Загрузка логотипа должна поддерживать прямую загрузку файла (PNG/SVG, макс. 10 МБ) вместо внешних ссылок.
  - **Удалить**: Телефон финдиректора, email финдиректора, ссылку на вложенные материалы.
- **CJM привязки к мероприятию (Customer Journey Map)**:
  - Из Мероприятия -> вкладка "Спонсоры": Пользователь видит бренды, привязанные *к этому конкретному мероприятию*.
  - Пользователь может нажать "Добавить спонсора" -> Найти и выбрать существующий бренд (в том числе импортированный).
  - Управление видимостью на сайте осуществляется через чекбокс "Выгружать на сайт" (`exportToWebsite`) в модальном окне спонсора мероприятия.
- **Особенности импорта**: При импорте старых данных (Legacy) логотипы автоматически проецируются на `logoUrl`, а бренд получает статус "Требует проверки" (Requires review) с отображением 100 элементов на странице по умолчанию.

### 2.2 Логика выгрузки Спикеров и Модераторов
- **Добавить**: Чекбокс "Выгружать на сайт" для Спикеров и Модераторов для переопределения поведения по умолчанию.
- **Логика выгрузки по умолчанию**: 
  - Модераторы/Спикеры автоматически попадают в выгрузку на сайт, ЕСЛИ они привязаны к мероприятию И их статус "Подтвержден" или "Предварительно подтвержден".
  - **Сортировка**: Сначала выводятся спикеры, а модераторы выводятся в самом конце списка.
- **Автоматический ЧБ Фильтр**: Все фотографии спикеров/модераторов должны отображаться в градациях серого (черно-белые) с помощью CSS-фильтров (`filter: grayscale(100%)`) во всей CRM и на публичных экспортах сайта, без изменения оригинальных загруженных файлов изображений.

### 2.3 Памятка спикера
- Реализовать пиксель-пёрфект HTML/CSS верстку на основе предоставленного скриншота-референса.

### 2.4 Интеграция с сайтом Tilda и извлечение верстки
- **Извлечение верстки**: Для парсинга и анализа верстки с `https://spring.upgrade.st/program` и `https://spring.upgrade.st/` будет использоваться сабагент (Subagent), чтобы сгенерировать адаптируемый HTML/CSS шаблон.
- **Подход к интеграции**: 
  - Поскольку Tilda полагается на визуальные блоки, оптимальный метод интеграции — предоставить публичные API эндпоинты (только для чтения) из нашей CRM (например, `/api/public/events/:id/website-data`).
  - Встроить пользовательский HTML/JS блок (Zero Block) внутри Tilda, который использует `fetch()` для динамической загрузки и рендеринга расписания, спикеров и логотипов брендов при загрузке страницы.

### 2.5 Синхронизация фотографий спикеров и миграция
- Добавлен рут `POST /api/speakers/sync-photos` на бэкенде DevStand для динамического импорта JSON-нагрузок со ссылками на фотографии из Tilda.
- Фотографии (около 188 штук) были успешно спарсены браузерным сабагентом в обход обфускации и LazyLoading Tilda. Установлен воркфлоу `/tilda_scraping_guide` для будущих нужд агентов.
- **Парсинг Брендов**: Импорт брендов из старой системы также осуществляется через скрипт-парсер, сохраняющий данные в базу.

### 2.6 Архитектура и UI превью интеграций
- **Единая точка входа**: Формирование кода для вставки, выбор нужного мероприятия и выбор макета (компоновки) виджета происходят **строго** и **только** в интерфейсе CRM в разделе "Интеграция с Tilda" (`/tilda-integration`).
- **Сайт-превью**: Скрипты предварительного тестирования виджетов (например, `test-tilda-standalone.html`) работают исключительно в режиме визуализатора (только для чтения - Read-Only). Они получают конфигурацию через URL-параметры (например, `?eventId=1&layout=new`) и **не должны** содержать выпадающих списков для выбора мероприятий, а также инструкций по копированию кода. Это исключает дублирование логики и отображение неактуальных "заглушек" данных на тестовых стендах.

## 3. Объем работ (Задачи)
1. **Изменения схемы базы данных**:

   - Добавить `marketSegments` (String[] или JSONB), `exportToWebsite` (Boolean), `logoFileUrl` (String).
   - Преобразовать `Sponsor` в глобальную сущность со связью "многие-ко-многим" с `Event` через связующую таблицу (например, `EventSponsor`), гарантируя сохранение исторических привязок к мероприятиям.
   - Обновить модель `SessionSpeaker`: добавить переопределение `exportToWebsite` (Boolean).
2. **Обновления Backend API**:
   - Миграции и эндпоинты CRUD для новой глобальной архитектуры Брендов.
   - Публичные API эндпоинты, адаптированные для Tilda (`/api/public/events/:id/website-data`), с учетом правил `exportToWebsite` и видимости на основе статуса.
3. **Обновления Frontend CRM**:
   - Переименовать пункт бокового меню в "Бренды".
   - Переработать UI Спонсоров Мероприятия для поддержки режимов "Выбрать существующего" vs "Создать нового".
   - Обновить Модальное окно редактирования Бренда, чтобы отобразить новые поля и логику загрузки файлов (с проверкой SVG/PNG до 10 МБ).
   - Добавить CSS `filter: grayscale(100%)` ко всем аватарам спикеров.
   - Разработать шаблон Памятки спикера.
4. **Миграция данных и Извлечение верстки (Subagent)**:
   - Запустить задачи сабагента для извлечения верстки Tilda и/или миграции старых данных по запросу.
