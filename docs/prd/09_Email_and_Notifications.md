# Архитектура: Email, Сброс пароля и Уведомления (Проектирование)

В этом документе описана архитектура для реализации email-рассылок, авторизации (сброса пароля) и триггерных уведомлений спикеров. Это концептуальный план, подготовленный для будущей реализации.

## 1. Сброс пароля (Password Reset)

Чтобы пользователь (менеджер) мог безопасно восстановить пароль по email, нам потребуется:

### Изменения в Базе Данных (Prisma)
Нужно добавить в модель `User` поля для хранения токена сброса пароля:
```prisma
model User {
  // ... существующие поля
  resetToken       String?   @unique @map("reset_token") @db.VarChar(255)
  resetTokenExpiry DateTime? @map("reset_token_expiry")
}
```

### Backend (NestJS API)
1.  **`POST /auth/forgot-password`**:
    *   Принимает `email` (в нашем случае `username`).
    *   Генерирует криптографически стойкий случайный токен (через `crypto.randomBytes`).
    *   Хэширует токен и сохраняет его в БД в поле `resetToken` вместе с `resetTokenExpiry` (например, на 1 час).
    *   Отправляет письмо со ссылкой: `https://upgrade-crm.com/reset-password?token=<unhashed_token>`.
2.  **`POST /auth/reset-password`**:
    *   Принимает `token` и `newPassword`.
    *   Хэширует полученный `token` и ищет пользователя в БД, проверяя срок годности (`resetTokenExpiry > now()`).
    *   Если валидно, обновляет пароль (через bcrypt) и очищает поля `resetToken` и `resetTokenExpiry`.

### Frontend (React)
*   **Страница `/forgot-password`**: Форма для ввода Email.
*   **Страница `/reset-password`**: Форма для ввода нового пароля. Берет `token` из URL-параметров.

## 2. Инфраструктура Email (Nodemailer)

Для отправки транзакционных писем (сброс пароля) и маркетинговых/уведомительных (спикерам) мы интегрируем библиотеку `nodemailer` в специальный `MailModule` на бэкенде NestJS.

### Провайдеры (SMTP)
В `docker-compose` или `.env` мы указываем данные SMTP-сервера. Можно использовать:
*   **Postmark / Resend / SendGrid**: Для массовых и надежных транзакционных писем.
*   **Корпоративный SMTP (Яндекс/Mail.ru/Google Workspace)**: Подходит для начальных этапов.

### Шаблоны писем
Для красивых HTML-писем лучше всего использовать шаблонизаторы, например, `EJS` или `Handlebars`, встроенные в NestJS, либо формировать красивые React-email шаблоны и рендерить их в строку на сервере перед отправкой.

## 3. Триггерные уведомления спикеров

В будущем менеджер в личном кабинете будет видеть задачи по обзвону и рассылке.

### Бизнес-логика триггеров
Вместо того чтобы менеджер вспоминал, кому и когда писать, система будет сама предлагать рассылки или автоматически отправлять их (Trigger Campaigns).

**Типы триггеров:**
1.  **"Напоминание о загрузке презентации"**: Если до мероприятия осталось 7 дней, а у спикера `hasPresentation = false`.
2.  **"Подтверждение участия"**: За 2 недели до мероприятия спикерам со статусом `review` отправляется письмо-опросник.
3.  **"Информационное письмо"**: Логистика, как добраться, где вход — за 1-2 дня до старта.

### Модель данных для рассылок
Для отслеживания того, было ли отправлено письмо, нам потребуется таблица:
```prisma
model SpeakerNotification {
  id        Int      @id @default(autoincrement())
  speakerId Int      @map("speaker_id")
  eventId   Int      @map("event_id")
  type      String   // 'reminder_presentation', 'logistics', etc.
  status    String   // 'sent', 'opened', 'failed'
  sentAt    DateTime @default(now()) @map("sent_at")

  @@map("speaker_notifications")
}
```

### Как это выглядит в CRM (Личный Кабинет Менеджера)
В личном кабинете менеджера появится вкладка **"Мои задачи / Уведомления"**:
*   Система покажет список: _"У этих 5 спикеров нет презентации, отправить напоминание?"_
*   Менеджер нажимает кнопку **"Отправить всем"**, бэкенд формирует очередь задач (например, через `BullMQ` и Redis) и плавно рассылает персонализированные письма.

Это позволит масштабировать работу, так как задачи будут генерироваться CRM-системой, а не в голове менеджера.
