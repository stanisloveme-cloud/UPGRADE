# PRD-13: Contract-Driven Development (Swagger + Frontend Typings)

## 1. Описание проблемы
В текущей архитектуре UPGRADE CRM фронтенд (React) и бэкенд (NestJS) жестко не связаны контрактами. Из-за этого изменение DTO (Data Transfer Object) на бэкенде или неверный тип данных, передаваемый фронтендом (например, строка `"400"` вместо числа `400` для вместимости зала), приводит к тихим ошибкам валидации 400 Bad Request. 

**Цель:** Исключить ошибки несовпадения типов данных между клиентом и сервером. Внедрить процесс, при котором изменение API автоматически отражается в TypeScript-типах фронтенда, и компилятор (`tsc`) не позволяет сбилдить проект, если формы отправляют неверные типы данных.

## 2. Предлагаемое решение

Мы внедрим подход **Contract-Driven Development**. 
В качестве "Единого источника истины" (Single Source of Truth) будет выступать Backend-код.

### Этап 0: Создание временной тестовой среды (Dev-2)
Чтобы не сломать текущий стабильный DevStand и Production, мы создадим полностью изолированную копию проекта (Dev-2) на предоставленном временном сервере. Все эксперименты со Swagger и кодогенерацией будут проходить исключительно там. Только после 100% успеха на Dev-2, изменения будут перенесены в основную ветку.

### Этап 1: Бэкенд (Генерация контракта)
1. Установим пакет `@nestjs/swagger` в `src/`.
2. Настроим `DocumentBuilder` в `main.ts` для автоматической генерации файла `openapi.json` из наших текущих контроллеров и DTO-классов.
3. При необходимости добавим декораторы Swagger (`@ApiProperty()`) в ключевые DTO (хотя встроенный плагин `@nestjs/swagger` в `nest-cli.json` может сделать 90% работы автоматически).
4. Настроим генерацию статического файла `openapi.json` при старте бэкенда или специальной командой `npm run api:export`.

### Этап 2: Фронтенд (Кодогенерация)
1. Установим кодогенератор, например `orval` или `openapi-typescript-codegen`, в `client/`.
2. Добавим скрипт в `client/package.json` (например, `npm run api:generate`).
3. Скрипт будет:
   - Считывать файл `openapi.json` от бэкенда.
   - Генерировать **TypeScript-интерфейсы** для всех моделей данных (Session, Speaker, Hall и т.д.).
   - Генерировать **React Query Хуки** (опционально) или просто типизированные Axios-функции (`getSession(id)`, `updateHall(id, data)`).
4. Все сгенерированные файлы будут сохраняться в папку `client/src/api/generated/`. Эту папку мы добавим в `.gitignore`, чтобы не засорять репозиторий (либо будем коммитить — обсудим в процессе, обычно коммитить надежнее для CI/CD).

### Этап 3: Рефакторинг (Без поломки текущего функционала!)
1. Мы НЕ будем переписывать весь проект зараз.
2. Мы переведем на новые сгенерированные типы только **один** компонент (например, `HallsModal` или глобальный `SessionDrawer`), чтобы проверить работоспособность, не сломав стабильную сетку залов. 
3. Мы убедимся, что сборка `npm run build` проходит успешно.

## 3. Артефакты и их расположение

- **Контракт:** `/openapi.json` (генерируется в корне сервера бэкенда).
- **Сгенерированный клиент фронтенда:** `/client/src/api/generated/` (содержит типы и Axios-обертки).
- **Конфиг автогенератора:** `/client/orval.config.js` или аналогичный.
- **Workflow инструкция для ИИ-Агентов:** `.agent/workflows/api_contract_update.md`.

## 4. Обучение ИИ-Агентов (Правила)

Чтобы агенты в будущем не ломали этот процесс, мы внесем жесткие правила:

1. **В файл `.antigravityrules` будет добавлено правило:**
   - *ID:* `frontend-api-contract`
   - *Правило:* `DO NOT manually write Axios requests or TypeScript interfaces for API responses in the frontend. ALWAYS use the generated methods from client/src/api/generated/. If the backend API changed, run 'npm run api:generate' in the client directory first.`
2. **Создание нового Workflow (`.agent/workflows/api_contract_update.md`):**
   - Это будет подробная Markdown-инструкция. Если пользователь просит "Добавь новое поле `X` в контроллер и выведи его на фронте", агент должен выполнить шаги строго по воркфлоу:
     1. Добавить поле в DTO бэкенда.
     2. Запустить команду создания/обновления Swagger `openapi.json`.
     3. Запустить `npm run api:generate` на фронте.
     4. Только после этого импортировать новый тип в React-компонент.

## 5. План действий (Roadmap)

1. **[Инфраструктура]** Настроить CI/CD пайплайн (или ручной деплой) на временный сервер `Dev-2`. Поднять там изолированные БД, Nginx и контейнеры.
2. **[Back-end]** Интеграция Swagger в NestJS, настройка `nest-cli.json` для вывода типов, генерация `openapi.json`.
3. **[Front-end]** Настройка `orval` в клиентской папке, пробная генерация типов из файла `openapi.json`.
4. **[Интеграция UI]** Точечная замена сырых вызовов `axios` в формах (типа Залов) на строго типизированные.
5. **[Регламент]** Добавление правила в `.antigravityrules` и создание воркфлоу-страницы для агентов.
6. **[Проверка на Dev-2]** Полный прогон `npx tsc -b` и тестирование интерфейса на новом сервере.
7. **[Релиз]** Перенос изменений на основной DevStand и Production.
