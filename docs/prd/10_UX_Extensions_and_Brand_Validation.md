# PRD 10: UX Разделы и Сценарии Пользователя (CJM)

## Введение
Этот документ содержит анализ пользовательских сценариев (Customer Journey Map - CJM) для дополнительных UX-разделов, унаследованных от старой системы, которые мы интегрируем в новую архитектуру (React + NestJS).

Цель документа — зафиксировать текущее (как на скриншотах) и желаемое состояние, чтобы ничего не сломать при разработке, а также добавить недостающие функциональные возможности (например, фильтры в сортировке спикеров).

---

## 1. Сортировка спикеров
**Скриншот:** `2026-02-14_22-45-41.png`
**Суть:** Единый список спикеров с возможностью ручного упорядочивания (стрелочки вверх/вниз) и переключателями (вероятно, активность/отображение на сайте).

### Текущий CJM (Как сейчас):
1. Продюсер или менеджер открывает "Сортировку спикеров".
2. Видит плоский список всех спикеров мероприятия без фильтрации по сессиям.
3. Чтобы поднять нужного спикера наверх, нажимает стрелочки вверх.
4. Менеджер делает скриншот экрана (или его части), чтобы отправить ТОП-20 спикеров коллегам/партнерам для согласования.

### Утвержденный CJM:
1. Менеджер открывает "Сортировку спикеров".
2. **Использует новые Фильтры** (по залу, по треку, по статусу подтверждения), чтобы сузить список релевантных спикеров.
3. Перетаскивает спикеров (Drag & Drop) или стрелочками меняет их порядок.
4. Выбирает нужных спикеров галочками слева (Мультивыбор).
5. Нажимает кнопку **"Экспорт в PDF/Изображение"** (или "Копировать список"), чтобы получить резюмирующий файл без необходимости делать костыльные скриншоты экрана.

## 2. Анонсы деловой программы
**Скриншот:** `2026-02-14_22-20-28.png`
**Суть:** Сводная таблица конференций/треков для контроля готовности материалов.

### Утвержденный CJM (Inline Editable):
1. Продюсер открывает раздел "Анонсы деловой программы".
2. Видит список всех конференций (без вложенных сессий, только верхний уровень).
3. Прямо в этой таблице продюсер имеет возможность изменять значения в ячейках: "Тип материала", "Дата готовности", "Статус", "Ссылка на материал". Эти поля являются служебными (служат для координации менеджеров) и не пересекаются напрямую с карточками сессий.
4. Продюсер делает выводы по тому, какие блоки еще не готовы к публикации, меняет им статусы и идет пинговать ответственных менеджеров.

## 3. Оповещение спикеров
**Скриншот:** `2026-02-14_22-21-08.png`
**Суть:** Журнал ручных коммуникаций со спикерами и отправки памяток.

### Текущий CJM (Как сейчас):
1. Координатор открывает "Оповещение спикеров".
2. Вводит имя в поиск (например, "Ольга") или использует фильтр "Поиск по сессии".
3. Включает тумблер "Только не оповещенные", чтобы отобразить тех, до кого еще не дошли руки.
4. Видит карточку спикера: его контакты (телефон, ссылка на Telegram, Email), сессии, в которых он участвует, и точную дату/время.
5. Рядом со спикером есть индивидуальная ссылка на его личный кабинет (памятку) `lk.upgradesrm.ru/...` с иконкой копирования.
6. Координатор копирует ссылку на памятку, переходит в Telegram/SMS и отправляет ее спикеру вручную.
7. Возвращается в интерфейс и щелкает переключатели (например: "Отправлено в телеграм").

### Утвержденный CJM (Ручной чеклист + Генерация Памятки):
1. Координатор открывает "Оповещение спикеров".
2. Вводит имя в поиск или использует фильтр "Поиск по сессии".
3. Включает тумблер "Только не оповещенные", чтобы отобразить тех, до кого еще не дошли руки.
4. Видит карточку спикера, где крупно выведены контакты (телефон, ссылка на Telegram, Email), сессии выступления и время.
5. Интеграцию с отправкой по Email пока откладываем на будущее (Feature Request). Задача менеджера — взять контакты (быстро скопировать телефон или кликнуть на TG), написать спикеру самостоятельно и скинуть ему уникальную ссылку на Памятку.
6. Рядом со спикером есть кнопка "Скопировать ссылку на Памятку" формата `upgradecrm.ru/speaker-memo/[hash]`. 
7. Координатор отправляет сообщение в мессенджер, возвращается в интерфейс и щелкает переключатель "Отправлено в телеграм" (сохраняя факт действия в CRM).

**Новая фича к этому разделу:** Внутри админки будет добавлен подраздел "Редактор шаблона памятки спикера" (WYSIWYG HTML редактор или простой конструктор). Этот редактор позволит модератору задать единый красивый HTML шаблон. В этот шаблон будут автоматически подставляться имя/фамилия и программа конкретного спикера при открытии публичной ссылки `upgradecrm.ru/speaker-memo/...` без создания отдельного домена 3-го уровня.

## 4. Загрузка презентаций на локальный диск
**Скриншот:** `2026-02-14_22-26-50.png`
**Суть:** Иерархическое дерево скачивания всех презентационных материалов.

### Текущий CJM (Как сейчас):
1. За день до мероприятия технический директор заходит в раздел.
2. Видит иерархическое дерево: **Дата -> Зал -> Трек -> Время сессии -> Спикер -> Файл**.
3. Нажимает синюю кнопку "Выбрать папку" вверху страницы.
4. Браузер скачивает все файлы презентаций на локальный диск по выбранному пути. (Функция File System Access API браузера Chrome на Windows).

### Утвержденный CJM (ZIP Архиватор):
- На сервере процесс будет собирать (и потенциально кэшировать) многоуровневый ZIP-архив с директориями `[Зал]/[Трек]/[Сессия]`.
- Учитываем технический риск: сборка архива может занимать продолжительное время в зависимости от объемов (размеры PDF/PPTX). На первом этапе реализуем синхронно, если будут таймауты — переделаем на асинхронную фоновую задачу с оповещением пользователя о готовности.

## 5. Проверка брендов (Brand Validation)
**Скриншот:** *Отсутствует в примерах (информация взята из текстового ТЗ)*
**Суть:** Инструмент заявлен в главном меню ("Проверка брендов").

### Утвержденный CJM:
1. Менеджер открывает вкладку "Проверка брендов" (в будущем можно переименовать в "Реестр спонсоров/партнеров").
2. Видит список компаний (Спонсоров/Участников), которые заведены в систему. **Это не связано со спикерами и их местом работы напрямую.**
3. Менеджер может создать новую карточку бренда, заполнив базовую информацию (Название, описание, телефон контактного лица, логотип и т.д.).
4. Менеджер нажимает кнопку "Сгенерировать ссылку для согласования".
5. Полученную ссылку отправляют представителю бренда.
6. Владелец данных/представитель бренда открывает публичную ссылку, визуально проверяет свою карточку и нажимает кнопку состояния: **«Подтвердить (Да, актуально)»** или **«Отклонить (Нет, есть ошибки)»**. Во втором случае представитель может оставить комментарий о требуемых правках.
7. В админке статус бренда меняется, и менеджер понимает, что информация верифицирована.
