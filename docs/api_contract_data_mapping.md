# API Contract & Data Mapping (Single Source of Truth)

Этот документ описывает **Контракт данных (API Contract)** для новой архитектуры приложения в рамках подхода Contract-Driven Development (CDD). 
Он является **единственным источником истины** (Single Source of Truth) при разработке и отладке UI-компонентов, описывая, какие поля ожидает бэкенд (NestJS + Prisma) и как должен их собирать и отправлять фронтенд (React + Ant Design).

> [!IMPORTANT]
> **Правило скрытого ID (Анти-дубликаторы):**
> При редактировании любой сущности в модалках (`ModalForm`, `DrawerForm`), фронтенд **ОБЯЗАН** принудительно внедрять `id` редактируемой сущности в итоговый payload формы. Если поле `id` не отрендерено скрытым инпутом (`<ProFormText name="id" hidden />`) или не добавлено вручную в `onFinish` (`submission.id = initialValues?.id`), бэкенд воспримет патч как новую сущность и создаст **дубликат**.

## 1. Event (Мероприятие)
Корневая сущность, объединяющая залы, расписание и настройки.

| Поле (DTO) | Тип БД (Prisma) | Описание / Правила |
| :--- | :--- | :--- |
| `name` | `String` | Обязательное. Название мероприятия |
| `startDate` | `DateTime` | Обязательное (ISO-строка с фронтенда). Принимается как строка, преобразуется в БД-дату |
| `endDate` | `DateTime` | Обязательное (ISO-строка с фронтенда) |
| `description` | `String?` | Опциональное описание |
| `location` | `String?` | Опциональное место проведения |
| `eventLogoUrl` | `String?` | Опциональная ссылка на логотип для памяток |

**Frontend Considerations:** На фронтенде редактируется через компонент `DatePicker.RangePicker`. Перед отправкой массив дат разбивается на `startDate = dates[0].toISOString()` и `endDate = dates[1].toISOString()`.

---

## 2. Hall (Зал)
Пространство, привязанное к конкретному Event, где проходят потоки (Track).

| Поле (DTO) | Тип БД (Prisma) | Описание / Правила |
| :--- | :--- | :--- |
| `eventId` | `Int` | Обязательное. Привязка к родительскому мероприятию |
| `name` | `String` | Обязательное. Название зала (например "Главный зал") |
| `capacity` | `Int` | Вместимость (по умолчанию 0) |
| `sortOrder` | `Int` | Сортировка для UI |

---

## 3. Track (Конференция / Трек / Поток)
Блок расписания внутри зала (Hall) на определенный день, ограничивающий вложенные сессии по времени. На старой платформе и в UI часто фигурирует под ярлыком **«Конференция»**.

| Поле (DTO) | Тип БД (Prisma) | Описание / Правила |
| :--- | :--- | :--- |
| `hallId` | `Int` | Обязательное. Привязка к родительскому залу |
| `name` | `String` | Обязательное. Название трека ("Ритейл стратегии") |
| `day` | `DateTime` | Обязательное. День проведения трека (передается как ISO Date строка) |
| `startTime` | `String` | Обязательное. Формат строго `HH:mm` (например "09:00") |
| `endTime` | `String` | Обязательное. Формат строго `HH:mm` (например "18:00") |
| `description` | `String?` | Опциональное описание (Аннотация) |
| `sortOrder` | `Int` | Порядок сортировки трека в зале |

**Frontend Considerations:** На фронтенде время собирается из `ProFormTimePicker.RangePicker` (массив Dayjs объектов `timeRange`). В хуке `onFinish` frontend обязан извлечь `.format('HH:mm')` и поместить их в `startTime` и `endTime`, после чего **УДАЛИТЬ** свойство `timeRange`, чтобы не засорять payload.

## 4. Session (Доклад / Дискуссия / Сессия)
Основная единица расписания. Представляет собой выступление, круглый стол или панельную дискуссию с участием спикеров.

| Поле (DTO) | Тип БД (Prisma) | Описание / Правила |
| :--- | :--- | :--- |
| `trackId` | `Int` | Обязательное. Привязка к треку |
| `name` | `String` | Обязательное. Название сессии |
| `startTime` | `String` | Обязательное. Формат `HH:mm` |
| `endTime` | `String` | Обязательное. Формат `HH:mm` |
| `description` | `String?` | Краткое описание / аннотация сессии |
| `speakers` | `Array` | Массив привязанных спикеров (см. структуру ниже) |
| `questions` | `Array` | Опциональные вопросы для обсуждения (Title, Body) |
| `briefings` | `Array` | Опциональные бриф-звонки (Datetime, Link, Comment...) |
| `managerId` | `Int?` | ID ответственного менеджера за сессию |
| `ignoreConflicts` | `Boolean?` | Управляет обходом бэкенд-предохранителя пересечения спикера |

---

## 5. Speaker (Спикер)
Глобальный профиль человека. Включает всю контактную и личную информацию.

| Поле (DTO) | Тип БД (Prisma) | Описание / Правила |
| :--- | :--- | :--- |
| `firstName` | `String` | Имя |
| `lastName` | `String` | Фамилия |
| `company` | `String?` | Место работы |
| `position` | `String?` | Должность |
| `email`/`phone`/`telegram` | `String?` | Контакты |
| `photoUrl` | `String?` | URL аватара (фото) |
| `isSponsor` | `Boolean` | Является ли спонсорским спикером (default=false) |
| `hasAssistant`/`assistantName`/`assistantContact` | `String/Bool` | Данные ассистента спикера |
| `bio`/`internalComment` | `String?` | Биография и внутренний комментарий для модераторов |

---

## 6. SessionSpeaker (Связующая таблица "Сессия-Спикер")
Хранит настройки участия конкретного спикера в строго одной сессии. Обновляется через вложенный массив `speakers` в DTO Сессии.

| Поле (SessionSpeakerDto)| Пример значения | Описание / Особенности |
| :--- | :--- | :--- |
| `speakerId` | `1054` | Обязательно: ID глобального спикера |
| `role` | `'speaker'` | Enum: `speaker` или `moderator` |
| `status` | `'confirmed'` | Enum жизненного цикла `review`, `confirmed`, `declined` и др. |
| `companySnapshot` | `'Ozon'` | **"Снапшоты" (ВАЖНО):** Фронтенд автозаполняет эти поля при выборе спикера, беря их из его глобального профиля. Это страхует от того, что спикер сменит работу через год, не разрушив исторические данные старых конференций. Обеспечивают независимость подписей. |
| `positionSnapshot` | `'CEO'` | Снапшот должности на момент выступления |
| `presentationUrl` | `https://s3...` | Ссылка на скачивание презентации (PDF/PPTX) |
| `presentationTitle` | `'Тренды рынка 2025'` | Заголовок презентации |
