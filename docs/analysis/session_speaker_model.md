# Архитектура: Модель данных Speakers и Сессий (SessionSpeaker)

В данном документе описывается, как устроена связь между спикерами и сессиями в UPGRADE CRM, и как работает механизм переноса исторических данных (snapshots).

## 1. Проблема: Миграция сотрудников
Спикеры со временем переходят из одной компании в другую и меняют должности.
Если бы компания и должность хранились **только** в глобальном профиле спикера (`Speaker`), то при изменении места работы спикера в 2026 году у него поменялась бы должность во всех прошлых сессиях (например, за 2024 год). Это нарушает историографию и аналитику.

## 2. Решение: Модель SessionSpeaker (Snapshot)
Для решения этой проблемы связь между `Session` и `Speaker` реализована через промежуточную таблицу (relation table) `SessionSpeaker`.

Сущность `SessionSpeaker` не просто связывает два ID, но и хранит **исторический срез (snapshot)** данных на момент конкретного выступления.

### Ключевые поля в `SessionSpeaker`:
- `speakerId` (Foreign Key)
- `sessionId` (Foreign Key)
- `companySnapshot`: Название компании, от лица которой спикер выступал **на этой конкретной сессии**.
- `positionSnapshot`: Должность спикера **на момент этой сессии**.
- `role`: Роль в рамках сессии (`speaker` или `moderator`). Различает спикеров от модераторов.
- `status`: Статус подтверждения (`confirmed`, `review`, `declined` и т.д.).
- `programThesis` / `newsletterQuote`: Тезисы и цитаты, специфичные для этого выступления.

## 3. Логика работы интерфейса (Автозаполнение)
При добавлении существующего спикера в сессию через интерфейс `SessionDrawer`, система автоматически "подтягивает" значения `company` и `position` из его текущего глобального профиля в поля `companySnapshot` и `positionSnapshot` привязки к сессии.

Администратор может отредактировать их для текущей сессии (например, указать другой отдел), при этом глобальный профиль спикера **не изменится**.

## 4. Логика импорта старых данных (Legacy Seed)
При импорте расписания из старой системы (например, JSON с `sales.upgradecrm.ru`), скрипт парсинга:
1. Ищет спикера по email или Имени+Фамилии. Если спикера нет — создает его глобальный профиль (`Speaker`), используя `company` и `job_title` из старой базы.
2. Привязывает спикера к сессии (`SessionSpeaker.create`).
3. **Обязательно** копирует `company` и `job_title` из старого JSON в поля `companySnapshot` и `positionSnapshot` для связи `SessionSpeaker`. Это гарантирует, что старые должности навсегда останутся закреплены за старыми сессиями, даже если позже администратор обновит глобальный профиль спикера.
4. Преобразует старые ID ролей. В старой базе `speaker_role_id = 1` означает Спикера, а `2` — Модератора.
5. Записывает вопросы к сессиям в отдельную таблицу `SessionQuestion`, используя формат заголовков вида `#1`, `#2` (извлекая только порядковый номер).
