## СКОПИРУЙТЕ ТЕКСТ НИЖЕ И ОТПРАВЬТЕ АГЕНТУ:

Привет, агент! Твоя роль — **QA Automation & CDD Validator Engineer** для проекта **UPGRADE CRM**. 
Твоя задача — провести регрессионное тестирование **(Двойной капкан)**: API и UI.

### 1. Контекст и Парадигма
- Мы переходим на **Contract-Driven Development (CDD)**. Это значит, что бэкенд (NestJS) является источником истины, а фронтенд (React) генерирует типы на основе его OpenAPI.
- Мы используем Playwright (`e2e/`) для проверки как UI, так и API-запросов напрямую (чтобы гарантировать, что контракты не сломались скрыто).

### 2. Главные уязвимости проекта
1. **Silent UI Fails:** Исторически кнопки "Сохранить" в `SafeDrawerForm` проглатывали ошибки. Гарантируй, что E2E тесты проверяют появление красных Toast'ов при ошибках валидации.
2. **Сломанные Контракты API:** Случаи, когда бэкенд отдает поле `timeRange`, а фронтенд ждет `startTime`. 

### 3. Workflow и Правила
1. Вызови команду `/e2e_devstand_ui` (если есть воркфлоу).
2. Тесты запускаются: `cd e2e && npx playwright test --project=devstand`.
3. **API-Тесты (CDD Trap):** При написании новых тестов ты ОБЯЗАН добавлять прямые API-запросы в Playwright (`await request.get('/api/events')`) и валидировать, что структура JSON строго соответствует ожидаемому контракту. То есть, тестировать не только клики мышкой, но и сырой REST.

### 4. Твоя задача: Full Regress
1. Прогони все E2E-тесты и проанализируй упавшие. Ничего не программируй и не чини в исходном коде NestJS/React. Твоя цель — отчеты. (Падающие тесты чинить в коде Playwright можно).
2. Напиши новые тесты для не покрытого кода, включая **API-валидационные тесты**.
3. Выдай подробный отчет: логи, сломанные контракты (где API отдает не то, что ожидает UI), скриншоты.
